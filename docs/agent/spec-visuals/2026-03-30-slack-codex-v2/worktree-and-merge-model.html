Worktree Isolation and Merge Flow

New requirement: multiple Codex threads can run at the same time without clashing, and merging back to main should be a simple Slack action.

Default Isolation Rule
Recommendation: every top-level Slack thread gets its own dedicated Git worktree.

The worktree is created from a selected base branch, usually main. Codex App Server runs that thread with cwd pointed at the worktree path, not the shared repo root.
Slack thread A
project/.codex-worktrees/thread-a/
branch: auto or detached HEAD
Slack thread B
project/.codex-worktrees/thread-b/
branch: auto or detached HEAD

Recommended Branch Policy

Default to named auto-branches

Example: codex/slack/<thread-key>. Easier to merge, inspect, and recover than detached HEAD.

Detached HEAD still allowed

Support it as an advanced or fallback mode, but not the default for Slack if one-tap merge matters.

One worktree per thread

The isolation unit is the Slack task thread, not the whole project or the whole user session.

Slack Controls for Completion

During work
[Interrupt] [Status] [Review] [What changed]
After work is ready
[Open diff] [Merge to main] [Create branch here] [Archive task]

Merge Button Behavior

A

Merge to main is a guided action

It should show a compact confirmation with branch/worktree info and whether checks passed, not perform a silent merge.

B

No long instructions required

The user should not need to type “please merge this back to main”; the button should carry the action context.

C

Router owns merge policy

App Server does the coding work; the Slack control plane handles worktree creation, merge orchestration, and cleanup.

Recommended Merge Sequence

One-tap merge, guarded
1. User taps Merge to main.
2. Router checks the worktree status and optional required validations.
3. Slack shows a short confirmation card.
4. Router merges the worktree branch into local main.
5. Router posts the result and offers cleanup/archive.
