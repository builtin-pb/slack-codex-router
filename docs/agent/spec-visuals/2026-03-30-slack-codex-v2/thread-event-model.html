Thread and Event State Model

Design the core around App Server threads and event subscriptions so richer progress can be added by rendering more event types, not by changing control flow.

Primary Mapping
Slack Identity
channel_id
thread_ts
slack_message_ts records
Codex Identity
app_server_thread_id
active_turn_id
latest_subscription state
Invariant: one Slack thread maps to one Codex App Server thread. Codex owns turns inside that thread.

Minimal Runtime States

Idle

Thread exists, no active turn running, waiting for a new user message or a control action.

Running

Codex is executing a turn and events are streaming to Slack.

Awaiting user input

Codex explicitly asked a question or presented options; Slack is waiting for a reply or Block Kit action.

Interrupted

The active turn was stopped; the thread remains valid and can continue with a new turn.

Turn Handling Rules

A

Use turn/start by default

Most Slack replies should become normal new turns in the existing Codex thread.

B

Use turn/steer for short corrective replies

If Codex is actively running and the user sends a brief refinement like “only touch docs”, steer instead of interrupt+restart.

C

Interrupt on explicit control

The Interrupt button always maps to turn/interrupt; the router does not silently cancel a turn just because a new reply arrived.

Persistence Model

Persist now

  • Slack thread -> Codex thread mapping
  • Current runtime state
  • Latest active turn id
  • Outstanding interactive prompt metadata
  • Rendered Slack message ids for update/dedupe

Do not persist as source of truth

  • Full Codex conversation transcript
  • Router-authored summaries of App Server state
  • Synthetic workflow stages
  • Duplicated skill inventories

Recovery Strategy

After process restart
1. Reload persisted Slack-thread mappings.
2. Reconnect to Codex App Server.
3. Re-read thread runtime state from App Server.
4. Rebuild subscriptions for active threads.
5. Resume rendering without inventing missing progress text.
