Slack Interaction Model

Goal: preserve normal Codex behavior while rendering it in Slack threads with interactive controls when Codex explicitly asks for them.

Thread Lifecycle in Slack
1. User starts a task
User: "Investigate the failing CI on main"

Bot: starts a Codex thread and subscribes to App Server events.
2. Codex streams normal progress
Bot: posts Codex-originated planning/progress text into the same Slack thread.

No router-authored preflight or workflow wrapper is inserted.
3. Codex asks a question or gives options
Bot: renders the question plus Block Kit buttons/selects when the prompt is structured enough.

User: can tap a choice or reply in free text.
4. User refines or interrupts
User: replies in-thread.

Router decides between turn/start and turn/steer based on whether Codex is actively running and whether the reply is a short corrective update.

Rendered Slack Message Types

Progress message

Plain in-thread text from Codex events. This is the default surface and stays close to CLI behavior.

Question card

Used when Codex asks for a short answer or explicit choice. Includes buttons or a select menu when the choices are clear.

Control strip

Persistent thread actions like Interrupt, Review, Status, and Show Diff. These are Slack-native controls, not pseudo-prompts sent to Codex.

Control Philosophy

A

Free text remains primary

The user can always just reply in-thread like normal Codex CLI conversation.

B

Interactive controls only when clear

Use Block Kit for explicit options, approvals, and thread controls, not for every Codex utterance.

C

No router rewording

The router should avoid paraphrasing Codex content except where Slack UI needs a control label or fallback text.
