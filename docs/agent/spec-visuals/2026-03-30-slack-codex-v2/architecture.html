Slack Codex Router v2: Runtime Architecture

Recommended direction: a Node-first Slack service built around Codex App Server, with your existing Slack bot credentials reused from the current .env.

Primary Data Flow
Slack Surface
Slack Channel -> Slack Thread -> Block Kit Actions
Top-level prompt starts a task
Replies become follow-up turns
Buttons answer Codex-originated questions and approvals
Router v2 Service
Slack event adapter
App Server client + event subscription
Slack renderer for Codex events
Thread state + message dedupe + recovery
Execution Layer
Codex App Server
Owns threads, turns, steer/interrupt, review mode, skills, collaboration modes, and user-input requests.

Boundaries

Slack stays thin

Slack does not invent its own workflow stages. It renders normal Codex behavior and relays user controls back into App Server.

App Server is authoritative

Planning, questioning, review mode, steering, and approvals come from Codex/App Server, not a router-authored preflight layer.

Reuse current bot setup

The v2 service reads the same Slack bot tokens and allowed-user values from your existing .env so you do not need a new Slack app.

Event Model

Core now

  • Start thread
  • Start turn
  • Interrupt turn
  • Render Codex questions as Block Kit
  • Persist Slack thread to Codex thread mapping

Layer later without redesign

  • Richer progress timelines
  • Skill visibility panels
  • Review artifacts
  • Thread history browser
  • More event types from App Server

Mapping Rules

A

Slack thread = Codex thread

Each Slack thread owns one App Server thread id.

B

Slack reply = new turn or steer

Normal replies create turns; quick corrective replies during execution can steer the active turn.

C

Buttons = structured answers

When App Server asks for input, Slack sends back the chosen answer instead of free-form router commands.
