Persistence, Recovery, and Rollout

Final design section: make v2 reliable enough for daily use, not just richer than the current router.

What the Router Persists

Persist

  • Slack channel + thread ids
  • App Server thread id
  • Active turn id and runtime state
  • Worktree path, branch, and base branch
  • Outstanding interactive prompt metadata
  • Slack message ids used for updates and controls

Do not duplicate

  • Full Codex transcript
  • Skill definitions
  • Synthetic summaries of Codex state
  • Independent router workflow stages

Recovery Model

After router restart
1. Load persisted Slack-thread records.
2. Restart or reconnect to Codex App Server.
3. Re-read each live App Server thread.
4. Re-subscribe to active threads and turns.
5. Rehydrate Slack controls without replaying fake history.

Failure Handling

Slack delivery failure

Do not lose Codex state. Retry Slack rendering separately from the App Server thread lifecycle.

App Server disconnect

Mark affected threads degraded, reconnect, and recover from App Server state instead of assuming the turn is gone.

Worktree setup failure

Fail before Codex starts coding. The thread should show a setup error and expose a Retry setup control.

Release Strategy

A

Breaking v2 is clean

No backward-compatibility burden for old thread/session state. Start from a new persistence schema.

B

Verify with real Slack, not just mocked tests

Before replacing the old router, validate the full flow against a real private channel and your existing bot tokens.

C

Reuse current .env

The existing Slack bot/app stays in place; configuration evolves, but bot setup should not need to be recreated.

Implementation Phases

Phase order
Phase 1: App Server client, Slack thread mapping, basic turn streaming
Phase 2: interactive prompts, control strip, explicit interrupt/review
Phase 3: worktree manager and merge-to-main flow
Phase 4: restart recovery, failure hardening, richer event rendering
